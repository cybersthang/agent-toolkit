## Problem

### Lỗ hổng cụ thể

`intent_router.py` (UserPromptSubmit hook) khi match prompt với pattern action-verb (`fix|refactor|sửa|tạo|...`) chỉ in 1 system-reminder gợi ý agent áp dụng `clarification-gate` skill (xem [`templates/claude/hooks/intent_router.py:485-492`](../templates/claude/hooks/intent_router.py#L485-L492) — output dạng `additionalContext`, exit 0). System-reminder này là **advisory**, không có cơ chế cứng nào kiểm tra response của agent có honor skill hay không.

Hệ quả: agent (đặc biệt model mạnh tự "judge" yêu cầu đủ rõ) có thể trả lời thẳng turn đầu mà bỏ qua hoàn toàn flow `UNDERSTANDING/ASSUMPTIONS/QUESTIONS`. DEV vừa trải nghiệm chính xác kịch bản này — và đã yêu cầu fix triệt để.

### Vì sao các enforcement hiện có không đóng được lỗ hổng

Đọc [`templates/claude/settings.json`](../templates/claude/settings.json) Stop chain có `evidence_audit.py` + `verify_lint.py` + `post_edit_verify_gate.py` + `debug_sentry.py`. Cả 4 hook này chỉ trigger khi response chứa **claim cần verify** (cite không có proof, claim done, traceback, …). Một response text-only dạng "tôi sẽ research X, đây là kế hoạch" KHÔNG chứa claim → vượt qua mọi Stop hook.

`invariant_guard.py` (PreToolUse Edit/Write) chỉ trigger khi agent **gọi tool sửa file** — không trigger trên text-only turn đầu.

Kết luận: cần 1 Stop-layer hook MỚI dedicated cho việc enforce *response shape* — khi `intent_router` đã suggest skill có contract output (như clarification-gate có 4 marker bắt buộc), Stop hook kiểm response chứa đủ marker; thiếu → deny.

### Vì sao không chọn Option C (deny ngay UserPromptSubmit)

Đã loại ở Q3 — sẽ phá `/implement` autonomy mode (vốn được thiết kế chạy không hỏi). Cần fix sao cho autonomy active = skip enforcer (mềm), không phải fix bằng cách phá UserPromptSubmit.

## Solution

### Architecture overview

3 thay đổi nguyên tử:

1. **`intent_router.py` — patch 2 chỗ**:
   - Khi `_matched_skills()` trả về danh sách có `clarification-gate` → trước khi `print(json.dumps(output)); return 0`, ghi thêm state file `.agent-toolkit/.last_intent_suggested.json`: `{ts, skills: [...], prompt_hash}`. Đây là tín hiệu cho Stop hook biết "turn này phải kiểm shape".
   - Trước short-prompt early-out (line 469), thêm `_capture_skip_clarification(workspace, prompt)` parse pattern `skip-clarification:\s*(.+)` → ghi `.agent-toolkit/.skip_clarification_next.json` `{ts, reason, ttl: 300}`. Pattern này song song với `_capture_bypass_invariant()` đã có (line 462-465).

2. **`clarification_gate_enforcer.py` — hook MỚI ở Stop event**:
   - Đọc envelope từ stdin (response text, stop_hook_active flag).
   - Recursion break: tôn trọng `envelope.get("stop_hook_active") == True` → exit allow ngay.
   - Skip path 1: `.autonomy_active.json` tồn tại + chưa expired → exit allow + log `bypass=autonomy`.
   - Skip path 2: `.skip_clarification_next.json` tồn tại + chưa expired → consume file (unlink) + exit allow + log `bypass=escape-token reason=<reason>`.
   - Skip path 3: `.last_intent_suggested.json` không tồn tại hoặc `"clarification-gate"` không nằm trong `skills` → exit allow silent (turn này intent_router không suggest skill này).
   - Skip path 4: state file > 600s tuổi (turn đã qua từ lâu, không phải turn ngay sau) → exit allow + cleanup file.
   - Shape check: grep response text cho 4 marker literal `UNDERSTANDING`, `ASSUMPTIONS`, `QUESTIONS`, `Searched:`. Thiếu marker nào → enforce theo `get_enforce_mode(workspace, "clarification_gate_enforcer")`:
     - `warn` → in `[clarification-gate-enforcer] warn: missing markers: <list>` ra stderr, exit 0.
     - `block` → in cùng message + exit 2 (Claude Code convention deny Stop).
   - `AGENT_TOOLKIT_STRICT=1` override mode → luôn block.

3. **`templates/claude/settings.json` — wire enforcer vào Stop chain**:
   - Thêm entry MỚI dưới `Stop`, vị trí SAU `evidence_audit.py` và TRƯỚC `debug_sentry.py` (để evidence-audit xử lý claim-without-evidence trước — vì nếu evidence-audit đã deny thì enforcer không cần chạy; debug_sentry chạy cuối cùng giữ pattern hiện tại).
   - Timeout 5s (hook chỉ làm grep + file IO ngắn).

### State files cần thêm

| File | Writer | Reader | TTL | Mục đích |
|---|---|---|---|---|
| `.agent-toolkit/.last_intent_suggested.json` | `intent_router.py` | `clarification_gate_enforcer.py` | 600s | Đánh dấu "turn này được suggest skill X" |
| `.agent-toolkit/.skip_clarification_next.json` | `intent_router.py` | `clarification_gate_enforcer.py` (consume single-use) | 300s | Escape token DEV gõ tay |

Cả 2 file đặt cạnh các state file đã có (`.bypass_next_edit.json`, `.autonomy_active.json`, `.stop_audit_count.json`) → cùng convention `.agent-toolkit/.<purpose>.json`.

### Marker contract (cố định, không config được)

4 marker literal phải có trong response để pass shape check:

| Marker | Lý do bắt buộc |
|---|---|
| `UNDERSTANDING` | Bằng chứng agent đã đọc prompt + paraphrase (chống vibe-translate) |
| `ASSUMPTIONS` | Bằng chứng agent đã list assumption rõ ràng (chống ngầm hiểu sai scope) |
| `QUESTIONS` | Bằng chứng agent đã hỏi DEV trước khi act (chống tự xử lý) |
| `Searched:` | Bằng chứng đã search trước khi hỏi (chống hỏi câu mà code đã có answer — yêu cầu cứng của clarification-gate skill) |

Không làm regex flexible (case-insensitive / synonym) — sẽ false-allow. Per Q3 confirmation, marker phải khớp đúng literal.

## User Stories

### Story 1 — DENY khi shape thiếu marker

Là DEV gửi prompt có verb mutate state ở turn đầu session (vd: "fix dashboard groupby"), tôi muốn agent PHẢI emit response có đủ 4 marker UNDERSTANDING/ASSUMPTIONS/QUESTIONS/Searched: — nếu thiếu marker, Stop hook DENY và in `[clarification-gate-enforcer] block: missing markers: UNDERSTANDING, Searched:`. Agent re-emit có shape đúng → mới được stop.

### Story 2 — ALLOW khi shape đầy đủ

Là DEV, khi agent đã emit đúng 4 marker, tôi muốn hook silent allow — không in noise, không thêm system-reminder. Hook chỉ "speak" khi có vi phạm.

### Story 3 — SKIP khi autonomy active

Là DEV vừa chạy `/go` hoặc `/implement` (file `.agent-toolkit/.autonomy_active.json` đang tồn tại + chưa expired), tôi muốn enforcer hoàn toàn bypass — vì autonomy mode là cam kết "DEV đã duyệt scope, agent execute không hỏi nữa". Hook log `bypass=autonomy` để audit.

### Story 4 — SKIP khi DEV gõ escape token

Là DEV, trong tình huống chat nhanh / emergency / hỏi đáp đơn lẻ, tôi muốn gõ `skip-clarification: quick chat về log error` trong prompt để bypass enforcer đúng 1 lần. intent_router parse token → ghi state file TTL 300s; enforcer consume + unlink file + log reason vào `.hook_fire_log.json`. Lần tiếp theo (cùng session) phải gõ lại token mới muốn skip tiếp.

### Story 5 — NO-OP khi không match clarification-gate

Là DEV gửi prompt thuần lookup ("file X ở đâu") không khớp regex action-verb, tôi muốn enforcer im lặng hoàn toàn — không đọc response, không emit gì, không tốn cycle. Logic: nếu `.last_intent_suggested.json` không tồn tại hoặc không chứa `clarification-gate` → exit 0 ngay.

### Story 6 — Tuân enforce_mode + STRICT env

Là maintainer agent-toolkit, tôi muốn enforcer dùng `get_enforce_mode()` infra đã có (line 274-309 của `_common.py`): per-hook mode trong `enforce_mode.json`, default fallback, và `AGENT_TOOLKIT_STRICT=1` env var override → block. KHÔNG tạo cơ chế config riêng cho hook mới này.

### Story 7 — Installer wire hook đúng vị trí Stop chain

Là DEV consumer (vd: project NAKIVO) chạy `setup.py update`, tôi muốn `settings.json` downstream được re-install với entry `clarification_gate_enforcer.py` ở Stop chain TẠI vị trí: SAU `evidence_audit.py` (xử lý claim-without-evidence trước), TRƯỚC `debug_sentry.py` (giữ pattern hiện tại — debug_sentry là layer cuối). Test smoke: `test_stop_chain_interactions.py` verify ordering.

## Implementation Decisions

### D1 — Vì sao Stop hook MỚI thay vì sửa `evidence_audit.py`?

Đã cân nhắc thêm layer 4 vào `evidence_audit.py` (file đã có 3 layer per comment line 2-40). LOẠI vì:
- `evidence_audit.py` đã 12378 bytes — thêm logic = vi phạm complexity_sentinel.
- 2 concern khác nhau: evidence-audit kiểm *claim-có-evidence-không*; enforcer kiểm *shape-contract*. Tách file giúp test riêng + revert riêng.
- Pattern matches `invariant_guard.py` (also dedicated Stop hook for orthogonal concern).

### D2 — State file vs envelope inspection

Cân nhắc: enforcer có thể tự parse lại prompt user từ envelope thay vì đọc state file do intent_router ghi. LOẠI vì:
- Stop event envelope KHÔNG chứa user prompt vừa rồi (chỉ có `stop_hook_active`, `response_text`).
- intent_router đã có sẵn regex `_matched_skills()` — duplicate logic = drift risk.
- State file pattern đã được `bypass-invariant` chứng minh hoạt động (line 408-441 của intent_router.py + invariant_guard.py).

### D3 — TTL chọn 600s cho `.last_intent_suggested.json`

Đủ cho agent suy nghĩ + tool calls + emit response (mức bình thường ~30-180s, edge case research lâu ~600s). > 600s → coi là state cũ, không apply cho turn sau (tránh false-deny khi DEV chat tiếp tục cùng session sau lâu).

### D4 — TTL 300s cho `.skip_clarification_next.json` (giống bypass-invariant)

Same pattern. Đủ cho agent reach Stop trong cùng turn, không leak sang session sau.

### D5 — Marker 4-cái không config được

Per Q3 confirmation: làm flexible/synonym = false-allow. Hard-code 4 marker literal — nếu skill clarification-gate đổi contract sau này, sửa hook luôn (cùng commit, có ADR).

### D6 — Version bump 0.12.3 → 0.13.0 (MINOR)

Theo SemVer + agent-toolkit convention (xem [`CHANGELOG.md`](../CHANGELOG.md) line 1-30): new hook + new feature surface (escape token) = MINOR. Không phải breaking (downstream chỉ thấy hook mới được thêm, có thể opt-out qua `enforce_mode.json` `clarification_gate_enforcer: warn`).

### D7 — `setup.py update` re-install xuống NAKIVO

Sau khi land upstream + test pass, chạy `/home/voducthang/agent-toolkit/setup.py update /home/voducthang/NAKIVO` để NAKIVO nhận hook mới + entry settings.json mới. KHÔNG hot-patch local NAKIVO trực tiếp (Q2=a).

### D8 — Default `enforce_mode` cho hook mới = `block` (NOT `warn`)

Chốt ở /clarify Q1=(a). `enforce_mode.example.json` template (upstream) phải có entry:
```json
"per_hook": {
  "...existing 5 hooks (warn)...",
  "clarification_gate_enforcer": "block"
}
```
**Lý do**: tuân constitution Principle 5 — "BLOCK > WARN. Never demote severity to keep a flow green". Nếu default = `warn` thì fix này vô nghĩa (vẫn để agent skip turn đầu, đúng kịch bản bypass DEV vừa gặp). Consumer muốn nới → tự set `"clarification_gate_enforcer": "warn"` trong `enforce_mode.json` của họ.

**Diff với 5 hook hiện tại**: 5 hook kia là *advisory* (spec_first_guard, implement_notes_gate, …) — block sẽ break dev flow ở phase non-critical → đúng là nên `warn`. Hook MỚI này là *contract enforcement* — block là default đúng.

**Cần `/adr-add`** sau /clarify để ghi nhận policy "new contract-enforcement hooks default `block`, new advisory hooks default `warn`" — phân biệt rõ 2 class hook, tránh argue lại lần sau.

### D9 — Escape token format = `skip-clarification: \S{8,200}`

Chốt ở /clarify Q2=(a). `_patterns.py` thêm constant:
```python
SKIP_CLARIFICATION_RE = re.compile(r'\bskip-clarification:\s*(\S{8,200})\b', re.IGNORECASE)
```
- Reason group 1 → ghi vào `.skip_clarification_next.json` (`reason` field).
- Reason < 8 ký tự non-whitespace → regex fail → KHÔNG tạo state file → enforcer chạy bình thường (DEV không bypass được bằng `skip-clarification: x`).
- Reason > 200 → cũng fail (tránh paste paragraph làm log nặng).
- 8 ký tự đủ ép 1-2 từ có nghĩa (`quick chat`, `urgent fix`, `debug log` đều ≥ 8 char).

**Lý do** (per /clarify Q2 search): không có ADR/canonical_decision tiền lệ; chọn `\S{8,200}` để tương thích Principle 5 (audit-able) — DEV đọc lại `.hook_fire_log.json` 1 tuần sau vẫn hiểu vì sao skip.

Reject alternatives:
- (b) "no limit" — bị loại vì agent có thể tự bypass `skip-clarification: ok` không ai biết tại sao.
- (c) "4-enum fixed" — bị loại vì cứng nhắc, edge case không cover được.

### D10 — Self-resolved decisions (no DEV gate)

Các quyết định dưới đây được resolve ở /clarify bằng 5-layer self-resolve, KHÔNG hỏi DEV:

- **Marker contract cứng (Q-OPEN-3 closed)**: 4 marker `UNDERSTANDING/ASSUMPTIONS/QUESTIONS/Searched:` literal, không cho config-override ở v0.13.0. *Verified qua D5 + ADR-005 (deterministic-only).*
- **acceptance_evals shape (Q-OPEN-4 closed)**: 7 entries dùng `grader: code` + `layer: hook` (us7 dùng `layer: installer`) + `probe.tool: pytest` + `probe.args.target: tests/test_clarification_gate_enforcer.py::TestUsN`. *Verified qua pattern v0.7.0-spec-first-guard.md.*
- **prompt_hash ghi vào state file (Q-OPEN-5 closed)**: YES — sha256 8-char prefix vào `.last_intent_suggested.json` (~50 byte). *Verified qua spec proposal, trade-off chấp nhận được, không có rule ngược lại.*
- **verify_lint chain extension (Q-OPEN-6 closed)**: NO — verify_lint scope = lint Verify Report markdown, không overlap response shape contract. *Verified qua scope analysis.*
