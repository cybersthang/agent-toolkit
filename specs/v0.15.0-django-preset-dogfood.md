# Spec: Django preset (proof of stack-agnostic claim)

## Problem

Toolkit ships 3 presets — `generic`, `odoo-12`, `odoo-17`. Two of three
are Odoo. The third (`generic`) is a STUB (`description: GENERIC preset
— no project-specific defaults`).

Claim "stack-agnostic core" is **empirically untested**. v0.13.0
introduced framework-overlay machinery (`<file>.<framework>.json`
picker in `setup.py`); without a second stack actually exercising
overlay paths, regressions go silent.

## Goal

Ship a **Django REST preset** as the second concrete stack — chosen
because:

- Python ecosystem (reuses existing `postgres` + python lint MCP probes)
- Different code layout (`<app>/views.py`, `<app>/models.py`, no `addons/`)
- Different exception namespace (`django.core.exceptions.*`)
- Different test runner (`manage.py test` or `pytest-django`)

This forces the overlay machinery to ACTUALLY DIFFERENTIATE between
two stacks, exposing any hidden Odoo assumptions.

## Acceptance evals

| ID | Probe | Expected |
|---|---|---|
| AE-1 | `setup.py init <empty-dir> --preset django --yes --dry-run` | Plan shows `coverage_config.django.json`, `intent_map.django.json`, `verification.django.json`, `debug.django.json` installed as `.json` (no `.django` suffix in target) |
| AE-2 | None of the Odoo-flavored files appear in install plan (`coverage_config.odoo.json` etc.) | `grep coverage_config.odoo.json <plan-output>` returns 0 |
| AE-3 | `canonical_decisions.django.json` picked over `.generic.json` fallback | `grep django <installed-canonical-decisions>` matches |
| AE-4 | A 1-app sample Django project boots → `mcp__realdata_test__` MCP server starts (no Odoo-bin lookup error) | MCP `env_status` returns OK |
| AE-5 | Edit `<app>/views.py` → `auto_test_runner` runs `<app>/tests/test_views.py` not Odoo `run_module_test` | test invocation matches Django mapping |

## Touchpoints

### New preset + overlays
- `presets/django.json` (~ 40 LOC) — framework=`django`, language=`python`, language_version=`3.10`
- `templates/codex/canonical_decisions.django.json` (~ 80 LOC)
- `templates/agent_toolkit/coverage_config.django.json` (~ 25 LOC) — `**/<app>/views.py`, `**/<app>/models.py`, `**/<app>/api/**.py`, `**/<app>/services/**.py`
- `templates/agent_toolkit/intent_map.django.json` (~ 70 LOC) — replace `odoo-*` skills with `django-*` skills where applicable
- `templates/agent_toolkit/debug.django.json` (~ 20 LOC) — add `django.core.exceptions.*`, `django.db.utils.*` patterns
- `templates/agent_toolkit/verification.django.json` (~ 25 LOC) — drop `__manifest__.py` probe, add `manage.py check`

### Existing-file changes
- `templates/cursor/skills/django/` — net-new skill folder (analog to `templates/cursor/skills/odoo/`). Skills: `django-code-review`, `django-codebase-discovery`, `django-deterministic-answers`, `django-tdd`.
- `templates/cursor/rules/django/` — Django version-specific rules.
- `tests/test_installer.py` — add `django` to preset shape parametrize.
- `tests/test_snapshot.py` — add `django` to count parametrize.

### Out of scope
- Django Channels / async views (start with REST only).
- DRF specifics (just-Django default).
- Django <4 LTS (target 4.2 + 5.0).

## Risks

- Existing Odoo presets may have **implicit assumptions** uncovered
  during Django bootstrap. Each will become a discrete ADR.
- `realdata_test` MCP server was written Odoo-shaped; will need a
  Django variant or parameterization. Estimate: + 100 LOC in
  `templates/codex/mcp_servers/realdata_test_server.py` or a new
  `realdata_test_django_server.py`.

## Definition of done

- Toolkit installs into an empty Django project without manual fixes
- All 5 acceptance evals PASS
- README's "Presets shipping today" table grows from 3 → 4 rows
- Stack-coverage score in self-assessment goes 7 → **9 confirmed**
  (not just claimed)
