# Spec: Gap completeness gate — mechanical enforcement of "fix all gaps before claiming done"

## Problem

DEV's complaint (verbatim, 2026-05-08, captured in `feedback_exhaustive_analysis`):
> "sao cứ mỗi lần lại ra 1 cái mới thế Nghiên cứu 1 lần nghiên cứu sâu chứ"

Observed pattern in v0.13.x session (2026-05-24):
1. DEV: "đã làm xong toàn bộ chưa"
2. AGENT: response surfaces 3 GAP/BLOCKER + auto-fixes them → claims done
3. DEV (later): "đã làm xong toàn bộ chưa" (same question)
4. AGENT: response surfaces 3 NEW gap (different from round 1)
5. Loop continues — each "check" round uncovers fresh issues that should have been caught the first time

This is the **drip-feed anti-pattern**. Each review pass produces new findings instead of converging to "no open issues".

DEV's mental contract: *"After implement+review, AGENT must fix ALL gaps and blockers. If something cannot be fixed, escalate to DEV explicitly."*

Toolkit currently has:
- `feedback_exhaustive_analysis` memory — advice text, agent reads at session-start but no enforcement
- `gap-fix-cycle` skill — describes diagnose-patch-rerun loop, but agent must voluntarily invoke
- `/verify` autonomy — "Blocker → giữ autonomy ON, agent có quyền fix tự do" but doesn't FORCE the loop

Gap: **memory + skill + autonomy = advice tier only**. No Stop hook BLOCKs an agent that claims "done" while a previous turn surfaced unresolved gaps.

Precedent for the fix pattern: `feedback_no_ai_commit` memory → `git_guardrails.py` hook (memory tier → mechanical DENY tier). Same evolution needed here.

## Goal

Mechanical Stop-hook gate that:
1. **Tracks open gaps** across turns via state file `.agent-toolkit/.open_gaps.json`
2. **BLOCKs Stop** when agent claims `done | verified | xong | hoàn tất` while ≥1 gap entry has `status: open`
3. **Forces explicit resolution** — agent must mark each open gap as `fixed | deferred | cant_fix` before claim
4. **Bypassable** — DEV single-shot bypass via `bypass-gap-gate: <reason>` (≥8 chars, audit-logged)

Side effect: `intent_router.py` (UserPromptSubmit) detects when agent emits a GAP/BLOCKER list → auto-writes entries into `.open_gaps.json` → next turn agent sees state, knows pending work.

## User stories

- **US1**: As DEV, when I ask "đã làm xong toàn bộ chưa" after agent surfaced N gaps, agent CANNOT claim done with `N>0` open gaps. Agent must explicitly resolve each.
- **US2**: As AGENT, after I emit a numbered gap list (G1/G2/G3 + descriptions), the next turn's claim-done is blocked until I either fix each gap (auto-deleted from state when fixed) or explicitly mark `deferred`/`cant_fix`.
- **US3**: As AGENT, I can mark a gap "deferred" with reason: response includes `gap-defer: G2 <reason ≥ 8 chars>`. The state file flips that entry to `status: deferred`, hook allows the claim through.
- **US4**: As AGENT, I can escalate to DEV: response includes `gap-cant-fix: G2 <reason — what DEV must decide>`. Hook allows; surfaces this to DEV via stderr.
- **US5**: As DEV, single-shot bypass: prompt contains `bypass-gap-gate: <reason ≥ 8 chars>` → hook skips one turn (state preserved). Audit-logged in `.open_gaps.json` history.
- **US6**: As AGENT in `auto-chain` mode (autonomy active), the gate must NOT block mid-chain — `/implement` runtime resolves gaps iteratively per `/verify` retry. Gate fires only when agent attempts to STOP (final claim) with open gaps.
- **US7**: As AGENT, every `Edit`/`Write` in a turn auto-flips any gap whose `desc` matches an edit target to `status: fixed` (best-effort match).

## Out of scope

- **Auto-fix the gap itself**. The gate enforces RESOLUTION (fix/defer/cant_fix), doesn't auto-patch. `gap-fix-cycle` skill is for the fixing.
- **Cross-spec gap tracking**. State is workspace-scoped, not per-spec. Future v0.20 could split.
- **Visualization** of open gaps. CLI tool `python .codex/tools/gap_status.py` already exists (per intent_map.json — verify).

## State file schema

`.agent-toolkit/.open_gaps.json`:
```json
{
  "version": 1,
  "gaps": [
    {
      "id": "G1",
      "surfaced_ts": 1748100000,
      "surfaced_turn_hash": "abc123",
      "desc": "Default implement_notes.json config file MISSING",
      "status": "open",
      "resolution_ts": null,
      "resolution_reason": null
    }
  ],
  "bypass_history": [
    {"ts": 1748100500, "reason": "emergency-hotfix"}
  ]
}
```

## Bypass mechanisms (3 tiers)

1. **Per-gap defer** — `gap-defer: G<N> <reason ≥ 8 chars>` in response: flips one entry, hook allows through.
2. **Per-gap cant-fix escalation** — `gap-cant-fix: G<N> <reason>` in response: same as defer but surfaces stderr "DEV decision needed".
3. **Whole-gate single-shot bypass** — `bypass-gap-gate: <reason ≥ 8 chars>` in next prompt: skip one Stop check. Audit-logged.

## Risks

- **R1**: Hook over-triggers — every response with the word "GAP" surfaces false positive. Mitigation: `GAP_LIST_EMIT_RE` requires numbered list pattern `G<digit>` + description, not bare "gap" mention.
- **R2**: State file pollution across unrelated specs — agent emits gap for spec-A, later works on spec-B, hook still blocks. Mitigation: TTL — gaps older than 24h auto-expire to `status: stale` (treated as resolved for gate purpose, kept for audit).
- **R3**: Recursion — hook fires, agent's clarification response itself contains "done" → loop. Mitigation: same `stop_hook_active` env var check as other Stop hooks.
