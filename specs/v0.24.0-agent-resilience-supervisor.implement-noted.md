# Implement Notes — agent-resilience-supervisor (v0.24.0)

## §1 Scope deviations

1. **`_resume_state.py` lives in `templates/claude/hooks/`** (lib, underscore)
   not `tools/` (spec §3). Reason: `session_brief.py` (a hook) imports it like
   `_common`; the kill-switch + crash-wrapper tests skip `_`-prefixed libs.
2. **Implemented T3 (notify) before T2 (watcher)** — T2 depends on notify.
   tasks.md order was T2→T3; dependency-correct order is the reverse.
3. **No `requirements.txt`** created (repo has none). `psutil` documented as an
   optional dep with graceful degradation (transcript-mtime-only) instead.
4. **Touched `tests/test_hook_crash_wrapper.py`** (not in §3) to skip
   `_`-prefixed lib modules — required so the new lib doesn't fail an
   existing "all hooks must use run_main_safe" assertion.

## §2 In-transcript trade-offs

- **Pivot from external full-auto supervisor → read-only stall-watcher +
  notify.** Driven by the /clarify finding that DEV uses the VSCode-extension
  chat, which an external process cannot wrap. Read-only + notify works for
  BOTH extension and CLI; CLI additionally gets `--relaunch`. This was the
  central design decision (see spec `re_scoped` field + §5 D1/D2).
- **Detection is liveness/transcript-based, not wall-clock-only.** DEV
  explicitly rejected naive timeout (long legit tasks). Primary signal =
  transcript advancing (busy work keeps writing) + process-alive (psutil);
  wall-clock is only a backstop. For notify-only, false-alarms are cheap.
- **Relaunch cap semantics**: `relaunch_cap=10` = max 10 relaunch attempts;
  the 11th is NOT attempted — instead notify DEV. Confirmed by
  `test_cap_10_then_notify` (run_claude called exactly 10×).
- **Idempotency reuses R9 manifest** rather than a new checkpoint — the
  resume-brief excludes done/passed items so a resumed run never redoes work.
