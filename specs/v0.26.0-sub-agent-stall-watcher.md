# Spec — sub-agent-stall-watcher (v0.26.0)

## 1. Problem

Sau khi ship v0.24 (resilience supervisor) + v0.25 (parallel-subagent-guard),
**DEV phát hiện 1 gap**: nếu một sub-agent TRONG wave bị treo (529 cạn hoặc
hang), **không có lớp nào bắt được**.

Bằng chứng (đã document đúng trong chính spec đã ship):
- `v0.24` `§5 D11`: *"v0.24 watcher CHỈ theo dõi phiên Claude chính (1
  transcript = file `.jsonl` mtime mới nhất). Multi-agent/sub-agent
  monitoring để spec parallel-subagent-guard (spec 2) lo"*.
- `v0.25` `§7 Out-of-scope`: *"KHÔNG kết hợp với agent_supervisor.py (v0.24)
  ở phase 1; supervisor + guard là 2 concern riêng (resilience vs conflict)"*.

→ Sub-agent stall fall through 2 spec. Hậu quả: trong 1 parallel wave, nếu
1/N sub-agent treo, các sub-agent khác xong → DEV chỉ thấy wave "stuck"
mà không biết AI nào treo. Phải kill toàn wave + restart manually.

## 2. Solution

**Extend** `tools/agent_supervisor.py` (v0.24) với 1 mode mới
`--multi-transcript` (hoặc tự kích hoạt khi `.parallel_wave.json` exists):

1. **Discovery**: đọc `.agent-toolkit/.parallel_wave.json` lấy danh sách
   `zones[].agent_id`. Map mỗi `agent_id` → transcript file của sub-agent
   (cách mapping CHÍNH XÁC cần verify — Q1).
2. **Per-transcript stall check**: với mỗi transcript, áp logic stall của
   v0.24 (transcript mtime + stall_seconds + cooldown). Tier1 error-string
   + tier2 process-liveness vẫn dùng nếu khả thi.
3. **Notify enriched**: payload chứa `agent_id` của sub-agent treo +
   transcript path + idle_seconds. Tận dụng `notify.dispatch` v0.24
   (toast/log/SMTP/webhook — không code mới channel).
4. **Activation**: chỉ chạy khi `.parallel_wave.json` exists +
   `wave_done=false` + TTL chưa hết + autonomy active. Manifest clear → 
   multi-monitor tự dừng. v0.24 main-session watcher chạy SONG SONG độc lập.
5. **Notify-only Phase 1**: không auto-relaunch sub-agent (Agent tool là
   harness, model-only spawn — toolkit không re-invoke được). DEV nhận
   notify → quyết định kill wave / continue / re-spawn.

## 4. User Stories

- **Story 1**: Multi-transcript discovery từ manifest agent_ids.
- **Story 2**: Per-sub-agent stall check (mtime independent).
- **Story 3**: Notify identifies which sub-agent treo (agent_id + path).
- **Story 4**: Activation chỉ khi manifest active (zero friction ngoài wave).
- **Story 5**: Lifecycle integrate v0.25 (manifest clear → stop monitor).
- **Story 6**: Cross-platform Win + Linux/Ubuntu.
- **Story 7**: Notify-only Phase 1 (no auto-relaunch sub-agent).

## 5. Implementation Decisions

Chốt ở /clarify 2026-05-28 (Q1 verified qua claude-code-guide; Q2-Q7 self-resolved):

- **D1 (Q1 — transcript discovery)**: Phase 1 watcher monitor MỌI file
  `*.jsonl` trong `~/.claude/projects/<encoded-project-path>/` thoả:
  (a) `mtime > wave.created_ts` (= xuất hiện hoặc cập nhật sau wave emit),
  (b) PATH khác `find_active_transcript()` (= không phải main session đã
  được v0.24 single-watcher track).
  Verified qua claude-code-guide: sub-agents CÓ transcript riêng cùng dir,
  ghi incremental. KHÔNG bridge `agent_id` DEV chọn (vd "agent-a") ↔
  `<session-id>.jsonl` ở Phase 1 — notify nêu transcript path để DEV
  correlate. Mapping enrichment defer v0.27 (xem F1 follow-up).

- **D2 (Q2 — notify granularity)**: AGGREGATE — 1 notify mỗi cooldown
  liệt kê ALL transcripts đang stalled tại thời điểm tick. Tránh spam khi
  nhiều sub-agent treo cùng lúc; DEV xem 1 toast/email/log line là đủ.

- **D3 (Q3 — cooldown scope)**: PER-TRANSCRIPT cooldown (mỗi
  `<session-id>.jsonl` có timer riêng trong `last_notify_ts` dict). Vẫn
  per-transcript dù D2 aggregate — nghĩa là transcript A vừa báo, transcript
  B mới stall vẫn được include trong notify aggregate tiếp theo ngay khi
  detect, không bị cooldown của A chặn.

- **D4 (Q4 — stall threshold)**: Default dùng chung `stall_seconds` (180s)
  với main session. Optional override `subagent_stall_seconds` trong
  `resilience.json` cho DEV muốn ngắn hơn (vd 90s khi sub-agent thường
  ngắn). Nếu null → fallback main `stall_seconds`.

- **D5 (Q5 — CPU per-sub-agent)**: SKIP Phase 1 — chỉ transcript-mtime.
  Sub-agents thường in-process trong main claude → psutil khó tách thread
  riêng. Tránh phức tạp; v0.24 process-alive cho main session đã đủ tier1
  signal (main chết → tất cả sub-agent gone).

- **D6 (Q6 — coexist với v0.24)**: 1 SUPERVISOR process, 1 loop, 2 mode
  tích hợp. Mỗi iteration:
  1. `check_once(main_transcript, ...)` — v0.24 main logic (giữ nguyên).
  2. Nếu `.parallel_wave.json` active: `check_subagent_transcripts()` —
     discover + per-transcript stall check + aggregate notify.
  Không 2 process, không 2 loop. Đơn giản, lifecycle gắn 1 chỗ.

- **D7 (Q7 — notify channel)**: REUSE y nguyên `notify.dispatch` v0.24
  (toast/log/SMTP/webhook). Payload extends:
  - `kind`: `"main"` (v0.24 default) HOẶC `"sub-agent"` (v0.26 mới).
  - `wave`: tên wave (cho sub-agent).
  - `transcript`: full path (DEV mở để inspect).
  - `stalled_count`: số sub-agent stall trong aggregate (D2).
  Subject/title prefix `[sub-agent]` để DEV phân biệt main-stall vs
  wave-stall trong toast/email.

- **D8 (notify-only Phase 1, từ Story 7)**: Phase 1 KHÔNG auto-relaunch
  sub-agent (Agent tool = model-only spawn — toolkit không re-invoke).
  v0.24 main-session `--relaunch` giữ nguyên cho main; sub-agent: notify
  + DEV manually kill wave / continue / re-spawn.

## 6. Testing

## 7. Out-of-scope

- **KHÔNG auto-relaunch sub-agent** — Agent tool model-only; toolkit
  không re-invoke. Phase 2 future có thể notify DEV với khuyến nghị
  "DEV gõ /agent re-spawn ...".
- KHÔNG thay đổi v0.24 main-session watcher (chạy độc lập song song).
- KHÔNG thay đổi v0.25 conflict-guard logic (concern riêng).
- KHÔNG handle race "sub-agent treo nhưng wave_done được set tay" (edge
  case rare — assume DEV declare-done chỉ khi thật sự xong).
- KHÔNG track sub-agent transcripts ngoài active wave (vd 1 lần Agent
  tool đơn lẻ không qua wave manifest).
