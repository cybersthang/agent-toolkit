# v0.28.0 — read-only GitLab CI MCP server

## 1. Problem / DEV ask

> "Trong agent-toolkit cần thêm 1 MCP nữa để kết nối tới GitLab check cái
>  build, vì mỗi lần code xong build toàn lỗi."

After pushing, CI on GitLab goes red and the dev has to leave the editor,
open the pipeline, click into the failing job, and scroll the log to find
the error. The agent should be able to do that itself — fetch the failing
job's trace — so it can fix the code in the same loop.

## 2. Scope

In:
- A new MCP server `gitlab` exposing **read-only** CI inspection.
- 5 tools: `env_status`, `latest_pipeline`, `pipeline_jobs`, `job_trace`,
  `build_errors` (the one-call "why is it red").
- Installer wiring (copy + `.mcp.json` entry) via the existing generic
  `mcp_servers` / `start_<name>_mcp.py` path — **no setup.py change**.
- Credentials in `mcp.local.env.example`; opt-in via project `mcp_servers`.

Out (deliberately):
- **No write/trigger/retry/cancel** tools. Taking authoring CI actions
  would conflict with `git_guardrails` (AGENT does not perform
  state-changing remote actions without explicit DEV authority). A PAT with
  `read_api` is sufficient.
- No artifact download, no MR creation, no commit status posting.

## 3. Design

- Mirror `jira_server.py`: stdlib `urllib` only (dependency-free),
  `SimpleMcpServer` framework, `{{ENV_PREFIX}}_GITLAB_*` env from
  `.codex/mcp.local.env` via `start_gitlab_mcp.py`.
- Auth: `PRIVATE-TOKEN` header (GitLab PAT). Base `{URL}/api/v4`; URL
  defaults to gitlab.com and tolerates a trailing `/api/v4`.
- Project resolution: numeric id passthrough, or URL-encode a
  `group/sub/project` path; default from `{{ENV_PREFIX}}_GITLAB_PROJECT`,
  per-call `project` override.
- Robustness: `pipeline_jobs`/`build_errors` paginate ALL jobs (per_page=100,
  loop pages, cap 50) and filter status client-side (avoids GitLab `scope[]`
  array-encoding quirks across self-hosted versions; a >100-job pipeline must
  not drop a late failing job). Traces are tailed (last N lines,
  char-capped) since build errors live at the end and full logs blow context.
- `build_errors` excludes `allow_failure` jobs (they don't fail the build).

## 6. How to enable (consumer)

```jsonc
// <project>/agent-toolkit.config.json  (or a private preset overlay)
{ "mcp_servers": ["codebase", "postgres", "realdata_test", "gitlab"] }
```
```bash
# .codex/mcp.local.env
<PREFIX>_GITLAB_URL=https://gitlab.yourco.com   # or https://gitlab.com
<PREFIX>_GITLAB_TOKEN=<PAT with read_api>
<PREFIX>_GITLAB_PROJECT=group/your-project       # numeric id also works
```
```bash
python setup.py update <project>   # regenerates .mcp.json + copies server
# restart Claude Code / Cursor → tools: mcp__gitlab__build_errors, …
```
